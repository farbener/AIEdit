AI Edit

A Lightroom Classic plugin that sends a photo to an AI model (Claude or ChatGPT), gets back a full set of develop adjustments, and applies them — with a visual preview, presets, and one-click revert. Built for portraits, tuned for getting the subject into the light without wrecking the background.

subject target · 122–148
0 · blackluminance (0–255)255 · white
The core idea: measure the subject's brightness, not the whole frame, and aim it into a flattering band — while a dark backdrop stays dark.
Overview

What it is

AI Edit adds two commands to Lightroom Classic's Library menu. Point it at one or more photos; it renders a preview, measures a histogram, asks the model for develop settings tailored to that image (and any style note you give), and applies them. You can preview the result and keep or revert before committing.

It runs entirely inside Lightroom as a Lua plugin and talks to your chosen provider (Anthropic or OpenAI) directly from your machine using your own API key. No external services, no upload step, no separate app.

At a glance Works in Lightroom Classic only (the cloud "Lightroom" has no plugin SDK). Needs an API key for a cloud provider — Anthropic (Claude) or OpenAI (ChatGPT) — or runs fully local via Ollama with no key. Uses claude-sonnet-4-6, gpt-4o, or a local model (qwen2.5vl:7b) respectively. Edits are non-destructive — they're Lightroom develop settings you can reset at any time.

Who this is for

Photographers who shoot batches of similar frames (headshots, events, product) and want a fast, consistent first-pass edit they can fine-tune — and anyone curious how to drive the Lightroom SDK from an LLM. The troubleshooting and reference sections assume no Lua knowledge; the source reference is there if you want to extend it.

Quickstart

Install in five steps

From zero to a working plugin. This takes about three minutes.

  1. Unzip the plugin

    Unzip AIEdit.lrplugin.zip. You'll get a folder named AIEdit.lrdevplugin. Put it somewhere permanent — not your Downloads folder — for example Documents/Lightroom/Plugins/.

  2. Open the Plug-in Manager

    In Lightroom Classic: File → Plug-in Manager…

  3. Add the plugin

    Click Add, select the AIEdit.lrdevplugin folder, and confirm. It should appear in the list with a green status light. Click Done.

  4. Choose a provider & add your API key

    Select a photo in the Library grid, then run Library → Plug-in Extras → AI Edit Selected Photo… Near the bottom, pick your AI provider — Anthropic (Claude), OpenAI (ChatGPT), or Ollama (Local) — then paste the matching API key (cloud providers only; Ollama needs none). Keys are saved automatically, separately per provider.

    Where to get a keyAnthropic: console.anthropic.com → API Keys. OpenAI: platform.openai.com → API keys. Your key stays on your machine (in Lightroom's plugin preferences) and is sent only to the provider you selected.

  5. Run your first edit

    With a photo selected, keep Preview on, leave strength at 100%, and click Analyse & Apply. Review the rendered preview, then Keep or Revert. That's it.

Lightroom Classic only If you don't see Plug-in Manager under File, you're probably in the cloud "Lightroom" app, which has no SDK. This plugin requires Lightroom Classic on Windows or macOS.
Tutorial

Edit your first portrait

Let's walk through a real edit together — a portrait shot against a dark backdrop, the case the plugin is tuned for. By the end you'll understand every control in the dialog.

1 · Select and open

In the Library grid, click a portrait. Run Library → Plug-in Extras → AI Edit Selected Photo… The dialog opens with the photo's name shown at the top.

2 · Choose your controls

Here's what each control does and how we'll set it for this first run:

Why 100%?The model already targets a well-lit face using the histogram. Strength is passed to it as guidance — a higher number asks it to push its corrections harder. Leave it at 100% and only raise it if a specific image consistently comes out too flat. Asking for very high strength (e.g. 180%) is what caused early over-bright, green-tinted results.

3 · Analyse & apply

Click Analyse & Apply. Behind the scenes the plugin renders a preview, measures the histogram, and asks the model for settings. After a few seconds the edit is applied and the preview window appears.

4 · Read the preview

The preview shows the rendered, edited photo on the left, and on the right: the exact adjustments the model chose, plus its reasoning — for example why it lifted exposure, how much it pulled highlights to protect the white shirt, and how it kept the backdrop dark. This is your chance to judge before committing.

5 · Keep or revert

Happy with it? Click Keep. Not happy? Click Revert and the photo returns exactly to where it started — losslessly. Try both; revert is safe.

You just did a full round-tripYou analysed, previewed, and reverted — non-destructively. Now try it with a style instruction like "warm, slightly filmic" and compare. When you find a look you like, save it as a preset (next section) so a whole shoot can get the same treatment.

How-to guides

Common tasks

Short, goal-oriented recipes for specific jobs. For things going wrong, see Troubleshooting below.

Save a preset for a whole shoot

Open the dialog, set your style instruction and strength, then click Save current as preset… Give it a name (e.g. "Headshots — neutral"). It now appears in the Preset dropdown. For the rest of the shoot, just pick it from the dropdown and the fields fill in.

RecommendedSave portrait presets at 100% strength. If you have an old high-strength preset from testing, re-save it at 100% to avoid over-bright results.

Edit a batch of similar frames

Select all the frames in the grid first, then run AI Edit Selected Photo… The action button reads "Apply to N photos." Preview is single-photo only, so for batches it applies directly and shows a summary at the end. Each photo is analysed individually, so exposure adapts per frame.

Revert a photo to its original

Select the photo and run Library → Plug-in Extras → AI Edit: Revert to Original. This resets it to Lightroom's default — exactly like the Develop module's Reset button — so it works even on photos edited long ago. Works on a selection too.

Give the model a creative direction

Type a short style note in the Style instruction field — describe the look, not the sliders. Good examples: "warm golden-hour skin", "cool editorial, low saturation", "punchy and contrasty". The model still respects exposure/histogram discipline; the note shapes colour and mood.

Find out exactly what happened

Open the log at Documents/AIEdit_debug.log. It records every step: the histogram reading, what the model proposed, the exact adjustments applied, and (on preview revert) a check confirming the photo returned cleanly. Send this file along with any bug report.

Troubleshooting

When something looks wrong

Each entry lists the symptom, the cause, and the fix. Most issues are either strength asked too high, an out-of-date build, or the RAW-vs-JPEG rendering difference. The log at Documents/AIEdit_debug.log tells you which.

The edit is too bright / overexposed

Symptom: the face comes out blown-out or milky.

Cause: historically two things — adjustment strength asked too high (e.g. a 180% preset), or, on an off-centre subject, the metering reading the dark background as if it were the subject and over-raising exposure.

Fix: set strength to 100% and re-save any high-strength preset at 100%. The off-centre case is addressed in the current build by skin-tone metering plus a background-detection flag (see Key concepts), so the model exposes to the actual face. Confirm on the Histogram: log line that skinLum reflects the face and, if exposure still looks high, check whether centerReliable=false was correctly flagged.

An off-centre face is mis-exposed (too bright or too dark)

Symptom: a portrait where the subject sits to one side gets the wrong exposure, as if the background drove the decision.

Cause: the centre-patch metric lands on background, not skin. On its own it would mislead the exposure choice.

Fix: the current build meters a skin-tone luminance independent of composition and flags the centre patch as unreliable when it equals the frame average. Check the Histogram: log line: skinPct should be non-trivial (the face was found) and skinLum should match how bright the face actually looks. If skinPct is near zero (face turned away, heavy glasses/hair, unusual light), the plugin falls back to the visible image — give a style hint or adjust exposure manually for that frame.

The dark backdrop turns green or muddy grey

Symptom: a clean dark-blue or black backdrop becomes olive-green or a flat milky grey.

Cause: two things, both from pushing too hard — an over-aggressive white-balance correction tints the neutral backdrop, and large Shadow/Black lifts haul the dark background up into grey.

Fix: run at 100% strength. The current build's prompt specifically protects a dark backdrop (gentle shadow lifts, restrained white balance), so this should not happen at normal strength. If it persists, capture the log and the preview screenshot.

The reverted RAW looks flatter / darker than the camera JPEG

Symptom: after reverting, the NEF/CR2/etc. doesn't match the bright, punchy JPEG the camera produced.

Cause: this is not a bug. A RAW file has no "look" baked in — Lightroom renders it with a neutral profile, which is flatter than the camera's own JPEG (the JPEG has the manufacturer's contrast, tone and colour processing applied in-camera). Reverting returns the RAW to Lightroom's genuine default, which is correct.

Fix: to get closer to the camera look, change the profile in Develop → Basic → Profile from "Adobe Color" to "Camera Standard" (or the matching maker profile). That's a profile choice, separate from revert.

"Revert to Original" didn't return to the true original

Symptom (older builds): revert produced an intermediate, still-edited state.

Cause: earlier versions replayed a stored snapshot that could be stale or captured after the photo was already modified.

Fix: the current build restores the saved pre-edit snapshot (the photo's own develop settings from before its first AI edit). Make sure you're on v1.3.1. If a photo is stuck in a bad state from old testing, run Revert to Original once (or use Lightroom's own Develop → Reset).

Error: "Assertion failed: packed"

Symptom: an error dialog when reverting.

Cause: revert re-applied the whole getDevelopSettings() blob, which includes string-encoded mask/retouch data that applyDevelopSettings can't unpack.

Fix: resolved in v1.3.1 — revert (and the preview Revert button) now restore only the develop keys the plugin actually writes, never the full blob. Update to v1.3.1.

Error: "Yielding is not allowed within a C or metamethod call"

Symptom (older builds): error during revert or apply.

Cause: a yielding SDK call (like applyDevelopSettings) was wrapped in pcall; Lua 5.1 can't yield across that boundary.

Fix: resolved in the current build. Update to v1.3.1.

"No photo selected"

Symptom: the plugin reports nothing is selected.

Cause: no active photo in the Library grid.

Fix: click a photo (or several) in the grid first, then run the command. The plugin uses the current selection, or the active filmstrip photo if none is selected.

The API call fails or times out

Symptom: an error mentioning the request, or a 401 / authentication failure.

Cause: missing/invalid API key, no internet, or no API credit.

Fix: re-check the key in the dialog for the selected provider (no spaces). Confirm the key is active and funded — Anthropic at console.anthropic.com, OpenAI at platform.openai.com. Check the log line AIProvider: response length for what came back; an API error: line shows the provider's own message (e.g. "model not found" → edit the model string in AIProvider.lua). Verify your connection.

The preview image looks like the old (unedited) version

Symptom (rare): the preview seems not to reflect the edit.

Cause: Lightroom rebuilds previews lazily and can briefly serve a cached thumbnail.

Fix: the current build forces a fresh render before showing the preview. If you ever see it, click Keep/Revert and check the photo in Loupe view — that's always the ground truth.

Still stuck?Grab Documents/AIEdit_debug.log (it logs every step), note your plugin version and Lightroom Classic version, and include a screenshot of the preview. That trio resolves almost everything.
Explanation

How it works

AI Edit is a Lua plugin running inside Lightroom Classic. It never leaves your machine except for the single HTTPS call to your chosen provider. Here's the architecture, plainly.

Lightroom's SDK lets a plugin add menu commands, read a photo's metadata and rendered thumbnails, and write develop settings back. AI Edit uses exactly those capabilities: it renders a small JPEG of your photo, analyses it, asks the model what to change, and writes the result back as standard develop settings — the same ones you'd move by hand in the Develop module.

Because the output is just develop settings, everything is non-destructive and lives in your catalog. Nothing is baked into the file; you can reset at any time.

Explanation

The edit pipeline

What happens between clicking "Analyse & Apply" and seeing the result.

  1. Render & meter. The plugin asks Lightroom for two JPEG thumbnails: a 1024px one for the model to look at, and a 400px one it decodes in pure Lua to compute clipping, a skin-tone subject luminance, and a centre-patch luminance with a reliability flag (see Key concepts).
  2. Ask the model. The thumbnail, the histogram numbers, per-photo metadata (camera, lens, ISO, RAW vs JPEG), your style note and the strength guidance are sent to your provider’s model (claude-sonnet-4-6 or gpt-4o). The system prompt instructs it to trust the histogram over the 8-bit preview and to expose for the subject.
  3. Apply. The model returns a JSON object of adjustments. The plugin maps each to a Lightroom develop key and applies them in one write, after snapshotting the prior state.
  4. Preview / revert. For a single photo with preview on, it forces a fresh render and shows the result with Keep/Revert. Revert restores the exact pre-edit state.
Explanation

Key concepts

Metering the subject: three signals, different jobs

Getting exposure right means knowing how bright the subject is — which is harder than it sounds when the subject is off-centre or sits against a very dark or bright background. AI Edit measures three things and gives each a specific role:

  • Clipping / headroom (how much is blown to white or crushed to black) is authoritative. An 8-bit preview makes bright-but-recoverable areas look identical to truly clipped ones, so here the numbers are trusted over the visual.
  • Skin-tone luminance is the primary subject signal. The plugin classifies pixels in YCbCr colour space and averages the brightness of those that fall in the skin range — so it reads the actual face wherever it is in the frame. Neutral greys (clothing) and greens (foliage) fall outside the skin range and are ignored.
  • Centre-patch luminance is a fallback hint, valid only when it's actually on the subject. If the centre reading is about the same as the whole-frame average, the patch is reading background, not a subject — the plugin flags it and tells the model to ignore it for exposure.

The model then fuses these with what it can see: if the numbers say "dark" but the visible face looks fine, the metering is sampling background and exposure is left alone; if the numbers and the visible face agree the subject is dark, exposure is raised. The target is a natural ~48–58% of full scale (about 122–148 on 0–255), and exposure moves stay within ±0.5 EV unless both signals agree a larger move is needed. This is what stops an already well-lit, off-centre face from being blown out — the failure mode of pure centre metering.

How strength works

Strength is not a numeric multiplier applied in code. It's sent to the model as a textual instruction ("adjustment strength: N%"), and the model scales the intensity of its own corrections accordingly. 100% is normal; higher asks it to push harder. This keeps the corrections coherent — the model balances exposure, contrast and colour together rather than a blunt scalar inflating every value.

Why the feedback loop is diagnostic-only

An earlier version tried to re-measure the photo after editing and auto-correct exposure. Two SDK realities make that unreliable: Lightroom returns a stale cached thumbnail right after an edit, and predicting the result from exposure alone ignores the other tone adjustments and Lightroom's non-linear exposure curve. So re-measuring read wrong values and over-corrected. The loop now only logs a brightness diagnostic; the first pass — which targets the histogram directly — does the real work.

How revert works

When applying, the plugin saves the photo's prior develop settings. Both "Revert to Original" and the preview Revert button restore that snapshot — but only for the develop keys the plugin itself writes. Re-applying Lightroom's full settings blob fails (it contains packed mask/retouch data), and an empty-table "reset" is a no-op in the SDK, so restoring the plugin's own keys is the reliable path.

RAW vs JPEG

A RAW file carries more highlight/shadow headroom than its 8-bit preview shows, so recovery moves are safe and effective — the plugin tells the model this. It also means a reverted RAW renders flatter than the camera's JPEG, because the JPEG has in-camera processing the RAW doesn't. That's a profile matter, not an edit error.

Reference

Source modules

The plugin is plain Lua against the Lightroom SDK (Lua 5.1 — no goto, no native libraries). Each file has one job. Line counts are a snapshot of v1.4.0 and will drift as the source changes.

Info.lua23 lines · manifest

Plugin manifest. Declares LrToolkitIdentifier = "io.github.farbener.aiedit", LrSdkVersion 6.0, the metadata provider, version 1.3.1, and the two Library menu items: AI Edit Selected Photo… → AIEditDialog.lua and AI Edit: Revert to Original → AIEditRevert.lua.

AIEditDialog.lua801 lines · entry point

The main flow. Opens with an architecture docblock describing the request flow and SDK constraints. Builds the dialog, gathers thumbnails and metadata, calls the API, applies edits, and runs preview/revert and batch. Everything runs inside LrFunctionContext.postAsyncTaskWithContext (a task is required for thumbnails, the dialog, HTTP and write access).

Key internal functions:

computeEdits(photo, style, strength) — renders thumbs, builds the histogram, gathers metadata, calls the API; returns the proposed edits (plus the pre-edit subject luminance for diagnostics).

applyEdits(photo, edits) — snapshots the prior scalar settings once, maps the model's keys to develop keys, applies them in one write.

feedbackCorrect(photo, preLum, exposure) — diagnostic only: logs predicted vs target subject brightness. No auto-correction.

processOnePhoto(...) / runBatch(...) — single and batch execution with progress and summary.

getThumbnail(photo, w, h) / measurePhoto(photo, size) — render helpers (named timeout constants; the holdRef GC-retention subtlety is documented inline).

AIProvider.lua364 lines · API client (multi-provider)

Sends the request to the configured provider and parses the JSON adjustment object. Shared code builds the prompt, attaches the image (pure-Lua base64), strips markdown fences and handles errors; a small ADAPTERS table holds the only per-provider differences — endpoint, request body/headers, and how to extract the reply text. Anthropic uses /v1/messages with an x-api-key header and a base64 source block; OpenAI uses /v1/chat/completions with Bearer auth, a data: image URI, and JSON mode; Ollama (local) uses the OpenAI-compatible /v1/chat/completions on localhost:11434 with no auth header and no forced JSON mode. Models are pinned per provider in the MODELS table (claude-sonnet-4-6, gpt-4o, qwen2.5vl:7b) — edit those strings to change models. The reply parser extracts the JSON object directly (first { to last }), so it tolerates models that wrap output in markdown fences or omit the closing fence.

Adding a provider (e.g. Gemini) is one ADAPTERS entry plus one MODELS entry; no other file changes.

AIProvider.analysePhoto(thumbPath, style, meta, histogram, strength) → ok, editsTable.

Histogram.lua219 lines · metering

Analyses a JPEG and returns three kinds of signal: clipping/headroom (authoritative), a skin-tone luminance (the primary subject-brightness signal, found by classifying pixels in YCbCr), and a centre-patch luminance with a centerReliable flag that is false when the patch is reading background. Plus a text summary for the prompt. Histogram.analyse(path) → summaryString, statsTable (keys include meanLum, skinMeanLum, skinPct, skinReliable, centerMeanLum, centerReliable, clipHiPct, clipLoPct).

JpegDecode.lua422 lines · JPEG decoder

A from-scratch baseline (SOF0) JPEG decoder in pure Lua (Huffman, IDCT, 4:4:4 / 4:2:0 / 4:2:2, restart markers). Needed because the SDK can't load native imaging libraries. Verified against Python PIL during development. Does not support progressive or arithmetic coding (Lightroom thumbnails are baseline).

Presets.lua57 lines · presets

Named recipes (style + strength + flags) stored as JSON in plugin prefs. Presets.names(), Presets.get(name), Presets.set(...), Presets.delete(name).

Settings.lua63 lines · preferences

Persists the chosen provider, a separate API key per provider, and last-used dialog values via LrPrefs. getProvider()/setProvider(), getApiKey(provider)/setApiKey(provider, key), getLast()/setLast(...).

AIEditRevert.lua133 lines · revert command

The Revert to Original menu command. Restores the saved pre-edit snapshot for the develop keys the plugin writes (not the full settings blob, which would throw a "packed" error). Header explains the SDK gotchas.

AIEditMetadata.lua15 lines · metadata field

Declares the hidden preEditSettings metadata field used to store the pre-edit snapshot on the photo.

Log.lua42 lines · logging

Timestamped logging to Documents/AIEdit_debug.log. Log.reset() (truncate at load), Log.write(message).

dkjson.lua714 lines · bundled lib

Third-party JSON encode/decode (David Kolf's dkjson). Bundled because the SDK has no built-in JSON module.

Adjustment schema

The model is required to return a single JSON object — no prose, no code fences — with these keys. All are numeric except reasoning. The plugin maps each key to a develop setting (see the key map) and applies the values directly; it does not multiply them by the strength setting.

// every key numeric except "reasoning"
{
  "Exposure": 1.2,            // stops, -5..+5
  "Contrast": 10,
  "Highlights": -45,
  "Shadows": 25,
  "Whites": -15,
  "Blacks": 5,
  "Texture": 20,
  "Clarity": 10,
  "Dehaze": 0,
  "Vibrance": 12,
  "Saturation": 5,
  "ColorTempKelvin": 5400,  // white balance
  "Tint": 4,
  "HueRed": 0, … HueMagenta  // 8 channels
  "SatRed": 0, … SatMagenta  // 8 channels
  "LumRed": 0, … LumMagenta  // 8 channels
  "VignetteAmount": -10,
  "GrainAmount": 0,
  "Sharpness": 60,
  "NoiseReduction": 15,
  "reasoning": "Subject centre at 30% … lifted exposure +1.2 …"
}
Prompt disciplineThe system prompt tells the model to trust the measured histogram over the 8-bit preview, expose for the subject centre, protect a dark backdrop from white-balance shifts and heavy shadow lifts, and recover highlights decisively on RAW files. The strength value is included here as guidance, so the model scales its own corrections.

Changelog & known limits

v1.4.0 — current

  • Ollama (local) provider. Run a local vision model via Ollama's OpenAI-compatible endpoint (localhost:11434), no API key needed. The dialog hides the key field and shows the endpoint instead. Default model qwen2.5vl:7b. Anthropic and OpenAI unchanged.
  • More robust JSON extraction. The parser now reads the JSON object directly (first { to last }) instead of relying on matched markdown fences — local models sometimes wrap output in a fence and drop the closing one.

v1.3.1

  • Fixed "Revert to Original" doing nothing. The previous approach relied on an empty-table reset, which is a no-op in the SDK. Revert now restores the saved pre-edit snapshot (the photo's own develop settings from before its first AI edit).
  • Fixed "Assertion failed: packed" on revert. Revert and the preview Revert button now restore only the develop keys the plugin actually writes, instead of the whole getDevelopSettings() blob (which carries string-encoded mask/retouch data that can't be re-applied).
  • Cosmetic: provider-neutral plugin name and dialog subtitle; removed the website link from the Plug-in Manager page.

v1.3.0

  • Skin-tone subject metering. Histogram.lua now classifies skin pixels (YCbCr) and reports their luminance as the primary subject-brightness signal — so exposure targets the actual face regardless of where it sits in the frame. Neutral clothing and foliage are excluded.
  • Metering-reliability flag. When the centre patch ≈ the frame average, it's flagged as reading background and excluded from the exposure decision.
  • Exposure fusion in the prompt. Clipping numbers are authoritative; subject brightness comes from skin metering cross-checked against the visible image; exposure stays within ±0.5 EV unless both signals agree on more. Target band lowered to a natural ~48–58%. This removes the over-brightening of well-lit, off-centre faces.

v1.2.0

  • Multiple AI providers. Choose Anthropic (Claude) or OpenAI (ChatGPT) from a dropdown; API keys are stored separately per provider so you can keep both. Built as a clean adapter layer (AIProvider.lua) so adding another provider later is a one-entry change.
  • Provider-aware dialog: the key label and "get your key at…" hint follow the selected provider; choice and key persist between runs.
  • Renamed the API module ClaudeAPI.lua → AIProvider.lua and made the dialog title and messages provider-neutral.

v1.1.0

  • Visual preview with rendered image, adjustment list, reasoning, and Keep/Revert.
  • Named presets, last-used recall, and per-photo metadata context (camera, lens, ISO, RAW/JPEG).
  • Dark-backdrop prompt protection (gentle shadow lifts, restrained white balance).
  • True-reset revert (both menu command and preview revert) — no more stale-snapshot reverts.
  • All-scalar snapshots (fixes the "packed" assertion) and removal of the pcall that caused the yielding error.
  • Feedback loop changed to diagnostic-only logging after the post-edit re-measure proved unreliable.
  • Detailed step-by-step logging, including a post-revert verification check.
  • Source documentation pass: module headers, named constants, corrected comments for handoff.

Known limits

  • Lightroom Classic only — the cloud app has no SDK.
  • Visual preview is single-photo; batches apply directly with a summary.
  • A reverted RAW matches Lightroom's neutral render, not the camera JPEG (use a Camera Standard profile to close the gap).
  • Post-edit re-measurement isn't reliable in the SDK (stale previews), so exposure rides on the model's first pass.
  • The system prompt was tuned with Claude; OpenAI may need minor prompt re-tuning to match exposure discipline exactly.
  • The SDK can't add toolbar or filmstrip buttons — commands live in the Library menu.
